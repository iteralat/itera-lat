Glosario rápido

Para leer este reporte sin tecnicismos:

Error 500
El sistema se cae y muestra "error interno". La pantalla no abre para nadie.
Error 403
El sistema dice "no tenés permiso". Es lo correcto cuando alguien intenta entrar a algo que no le corresponde (no es una falla).
Script / JavaScript
El código que hace que la pantalla funcione en el navegador (que los botones respondan, que las cosas se muestren bien).
CSRF
Una protección para que una página externa no pueda ejecutar acciones (por ejemplo, un borrado) en nombre del usuario que está logueado.
Ruta / dirección
La dirección interna de cada pantalla (lo que aparece en la barra del navegador, por ejemplo /pacientes).

1 Resumen

Se aplicó el nuevo diseño a todo el sistema, se corrigieron problemas que ya venían de antes, se agregó el módulo de Alertas y se armó un panel de administración global (para un Super Admin). Además se dejó una demo con tres residencias y pacientes de ejemplo, para poder mostrar cómo se gestiona el sistema con más de una residencia a la vez —algo que antes no se había probado— y verificar que los datos de cada residencia quedan bien separados. Todo quedó respaldado con pruebas automáticas.

2 Problemas corregidos

Al revisar el sistema recibido se detectaron problemas. Se corrigieron los siguientes. Para cada uno: qué fallaba y qué riesgo evitaba corregirlo.

  1. Tableros de "médico" y "enfermero" daban error 500. No abrían para nadie. Sin corregir, esos roles no tenían pantalla de inicio funcional.
  2. La pantalla vieja de gráficos daba error 500 para todos los roles. Sin corregir, quedaba un acceso que siempre rompía; ahora redirige al módulo de gráficos nuevo.
  3. El sistema respondía "error de sistema" (500) cuando correspondía "no tenés permiso" (403). Daba la sensación de sistema roto ante un acceso denegado normal; podía disparar reportes de falla que no eran tales.
  4. El login cargaba con un script bloqueado por un control de seguridad mal configurado. Sin corregir, parte del código de la pantalla de ingreso no cargaba y quedaba con error.
  5. El logo de la barra superior no era clickeable. Detalle de navegación: ahora lleva al inicio.
  6. El primer ingreso del administrador caía en una pantalla vacía. Sin corregir, la primera impresión del sistema era una pantalla sin contenido; ahora entra directo a su pantalla útil.
  7. Accesos por dirección directa demasiado abiertos. Varias pantallas (usuarios, patologías, pacientes, archivos, configuración clínica) se abrían escribiendo la dirección a mano, incluso para roles que no debían verlas. Riesgo: exposición de datos clínicos a quien no correspondía. Se cerró por rol.
  8. Aislamiento entre residencias incompleto en seguimiento. Se podía acceder a un paciente de otra residencia por dirección directa. Riesgo: ver o editar datos de una residencia ajena. Se agregó control por residencia.
  9. Borrados y configuración clínica sin protección (CSRF). Riesgo: una página externa podía disparar un borrado o un cambio en nombre del usuario logueado. Se agregó la protección donde faltaba.
  10. Validación superficial de archivos subidos. Se aceptaban por extensión sin verificar el contenido real, y un archivo podía quedar guardado aunque fallara el registro en base. Riesgo: archivos falsos o huérfanos. Se agregó verificación de contenido y limpieza automática.
  11. El calendario mostraba turnos con datos como "sin datos". Confundía al leer configuraciones importadas. Riesgo: dar por vacío un día que sí tenía carga. Corregido.
  12. Las notificaciones mostraban textos técnicos confusos (por ejemplo "se eliminó '1' del turno"). Riesgo: registro de auditoría ilegible. Ahora se leen con nombres claros.

3 Rediseño aplicado

  • 26 pantallas migradas al nuevo diseño, una por una, preservando lo que ya funcionaba (guardar seguimientos, subir archivos, alta de pacientes, permisos, gráficos).
  • Marca unificada y centralizada: el diseño se armó primero con un color de marca violeta y luego se decidió pasarlo a verde (turquesa). Quedó todo unificado y el sistema quedó bien preparado para ajustar la marca a futuro.
  • Biblioteca de piezas reutilizables (botones, tarjetas, tablas, formularios, modales, estados vacíos) montada dentro del propio sistema.
  • Accesibilidad aplicada en todas las pantallas.

4 Funcionalidad nueva: módulo de Alertas

Desarrollo nuevo de punta a punta: avisos urgentes y accionables para el equipo (por ejemplo "cambiar dosificación a este paciente"), con campana en la barra superior, contador de pendientes, prioridad urgente/normal y flujo de "atender" con comentario de qué se resolvió.

  • Cada residencia ve solo sus propias alertas.
  • Los tres roles del equipo (gerontólogo, médico y administrador de la residencia) pueden crear, ver, atender, reabrir, editar y eliminar las alertas, para que nada quede trabado cuando falta algún integrante del equipo.
  • Blindado con pruebas por rol (crear, atender, reabrir, editar, eliminar).

5 Administración global

  • Panel de Super Admin nuevo: quien administra la plataforma ve el total de residencias, usuarios, pacientes y patologías, separado de la operación diaria de cada residencia.
  • Separación de experiencias: cada rol entra a su pantalla y su menú; cuando se miran datos de una residencia, la pantalla deja claro cuál es.
  • Roles y rutas ordenados: un rol viejo y ambiguo se unificó (con respaldo previo y sin poder dejar el sistema sin administrador); las rutas viejas (las direcciones dentro del sistema) siguen funcionando para no romper enlaces guardados.
  • Alta y edición de más de una residencia revisadas y funcionando, y el sistema quedó probado con tres residencias demo a la vez para ver cómo se separan los datos entre ellas.
  • Borrado seguro de residencias: se agregó una protección para que, si alguien intenta borrar una residencia, el sistema avise que se van a borrar todos sus datos y pida confirmar de forma explícita (escribiendo el nombre exacto).

6 Demo y garantías

  • Demo de 3 residencias (Los Arrayanes, Valle Sereno, Puentes del Limay) con datos ficticios, selector de usuarios en el ingreso, publicada en demo-jubilo.itera.lat.
  • 144 pruebas de aplicación + pruebas de navegación por rol, todo en verde.
  • Producción del cliente intacta.
